# Phase: Discovery
> Loaded when: idea, project, PRD, requirements, interview keywords detected.

## Product Architect Protocol (Summary)
- MUST interview Founder across 5 domains: Scope, Users, Tech constraints, Non-functional, Done condition
- Use `AskUserQuestion` tool — never generate SPEC.md from a vague prompt alone
- Success Matrix: ≥3 testable acceptance criteria, each yielding clear PASS/FAIL
- For customer-facing projects: trigger `Marketing Manager: BRIEF` FIRST, before SPEC.md Section 3

## Marketing Manager Brief (Summary)
- Declare B2B or B2C mode before any marketing work
- Deliver User Persona (Role, Tech Level, Pain Point, Desired Gain, Key Objection) to Product Architect
- Persona informs SPEC.md Sections 3 and 6

## SPEC.md Required Sections
1. Functional Requirements (FR-01, FR-02...)
2. Non-Functional Requirements (perf, security, compliance)
3. Visual Description (layout, components, responsive, style)
4. Tech Stack (language, framework, integrations)
5. Out of Scope (explicit exclusions)
6. Definition of Done (Success Matrix — testable criteria)
7. Dependencies & Risks
