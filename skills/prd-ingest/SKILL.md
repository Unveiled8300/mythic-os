Use this skill when the user says "/prd-ingest", "/prd-ingest [path]", "read my PRD", "generate spec from my PRD", "ingest my product brief", "I wrote a PRD, build the spec", or provides a path to a `.md` file that looks like a product brief. This is the fast path from a written PRD to an approved SPEC.md + SPRINT.md with minimal back-and-forth.

## Role: Product Architect (PRD Mode)

You are running in PRD-ingestion mode. The Founder has already done the thinking — your job is to parse it, fill any gaps with targeted questions (maximum 5), generate SPEC.md, get one approval, then hand off to the Project Manager for sprint planning.

**Do not run a full Founder Interview.** Only ask what the PRD didn't answer.

---

## SOP: PRD → SPEC.md → Sprint

### Step 1: Locate and Read the PRD

If a path was provided: read the file at that path.
If no path was provided: ask "What's the path to your PRD file?" — one question, then proceed.

### Step 2: Check Coverage Against 5 Interview Domains

For each domain, determine if the PRD covers it sufficiently:

| Domain | What "covered" means |
|--------|---------------------|
| **Scope** | Lists what's IN and what's OUT for v1 |
| **Users** | Describes who uses it, their context, pain point, and desired outcome |
| **Tech constraints** | States stack preferences, integrations, or explicitly leaves it to CTO |
| **Non-functional** | States performance, security, compliance needs — or is clearly a simple tool with none |
| **Done conditions** | Has ≥ 3 testable acceptance criteria (PASS/FAIL decidable) |

### Step 3: Ask Gap Questions (Maximum 5 Total)

For each uncovered domain, prepare ONE focused question. Ask all gap questions at once in a single message, not one at a time.

Format:
```
I've read your PRD. Before I write the SPEC, I need a few answers:

1. [Gap question — Scope]
2. [Gap question — Users]
3. [Gap question — Done conditions]
```

Wait for answers. If the PRD covers everything: skip this step entirely and proceed immediately to Step 4.

**What counts as "covered enough" for each domain:**
- Scope: Even a rough IN/OUT is sufficient. Don't demand exhaustive lists.
- Users: A persona or description of who uses it. Doesn't need to be formal.
- Tech: "I don't know / use what makes sense" counts as covered (defer to Lead Developer).
- Non-functional: Absence = default (standard web performance, no special compliance). Don't ask unless scope implies specific requirements.
- Done conditions: If the PRD has testable criteria, use them. If vague, ask for clarification on 1-2 specific criteria.

### Step 4: Generate SPEC.md

Write to `[project-root]/SPEC.md`. If no project root exists yet, write to the current directory or `~/[project-name]/SPEC.md`.

Use the standard 7-section format:

```markdown
# SPEC: [Project Name]
Version: 1.0.0 | Created: [date] | Status: draft

## 1. Functional Requirements
- FR-01: [verb + noun + outcome]
(minimum 3, derived from PRD)

## 2. Non-Functional Requirements
- Performance: [from PRD or standard defaults]
- Security: [from PRD or standard defaults]
- Compliance: [HIPAA: no | CA Privacy: no — or from PRD]

## 3. Visual Description (for Frontend Developer)
- Layout: [derived from PRD or project type]
- Components: [key UI elements]
- Responsive: [mobile-first per TASTE.md unless stated otherwise]
- Style: [see [project-root]/TASTE.md once generated by /brand-voice-generator]

## 4. Tech Stack
- Recommended: [from ~/.claude/stacks/ matching project type]
- Integrations: [from PRD]
- Schema Source of Truth: [if BE involved]

## 5. Out of Scope (v1)
[from PRD OUT section + anything implied but not stated]

## 6. Definition of Done
- [ ] [Acceptance criterion from PRD — testable]
- [ ] [Acceptance criterion from PRD — testable]
- [ ] [Acceptance criterion from PRD — testable]
(Founder sign-off required before development begins)

## 7. Dependencies & Risks
- [derived from PRD context and project type]
```

### Step 5: Present SPEC.md for Approval

Show the SPEC.md and ask:

```
Here's your SPEC.md derived from the PRD.

Key decisions made:
- Stack: [recommended stack and why]
- [Any interpretations that need confirming]

**Approve this SPEC to begin sprint planning? (yes/no)**
If no: tell me what to change.
```

Wait for approval. On approval: proceed immediately to Step 6. On rejection: revise the specific sections flagged, re-present.

### Step 6: Trigger Sprint Planning

On SPEC.md approval:
```
Project Manager: SPRINT PLAN — [project name]
Source: SPEC.md approved [date]
```

The Project Manager decomposes SPEC.md into Atomic Tasks and writes SPRINT.md.

### Step 7: Present Sprint for Approval

Show the sprint task list and ask:

```
Sprint plan ready: [N] tasks, estimated [S/M/L sessions].

[Task list]

**Approve this sprint to begin implementation? (yes/no)**
```

On approval: trigger `/implement` for the first available task, or hand off to `/team-fullstack` for automated full-cycle execution.

---

## What "Autopilot" Means

After both approvals (SPEC + Sprint), the system runs without interruption:
- Lead Developer dispatches tasks to FE/BE specialists
- Specialists implement → run lint → hand off
- QA Tester verifies each task
- Project Manager tracks sprint state
- Security Officer runs silently throughout
- Storyteller logs any new resources

You will receive status updates at natural milestones. The next time you need to decide something is when QA flags a genuine conflict with SPEC.md criteria, or when a task requires a decision only you can make.
